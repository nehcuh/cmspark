# Dynamic Workflow — Prompt Chain 元工作流

## 核心思想

每个复杂任务都应该拥有自己的"执行引擎"。v2 将能力从"直接写代码"升级为**"设计工作流，生成 Claude Code 提示词，让 Claude Code 执行"**——即元工作流（meta-workflow）。

## 何时使用

| 任务特征 | 推荐模式 | 说明 |
|:---|:---:|:---|
| 需要修改多个文件 | ✅ 扇出并综合 | 多个 subagent 并行处理 |
| 质量要求高（安全/审计） | ✅ 对抗式验证 | 双重把关 |
| 不确定最佳方案 | ✅ 锦标赛 | 多方案竞争选优 |
| 任务量不确定 | ✅ 循环至完成 | 直到满足停止条件 |
| 大量独立子任务 | ✅ 扇出并综合 | 互不干扰并行执行 |
| **大规模跨会话项目** | ✅ **Prompt Chain** | **生成 Claude Code prompt chain** |
| 简单单文件修改 | ❌ 不使用 | 直接修改即可 |

## 七大模式

### ① 扇出并综合（Fan-out & Synthesize）
**适用**：大量独立子任务、多模块审计、并行修改
**流程**：拆成 N 个小步骤 → 每个步骤跑一个 subagent → barrier 等待所有完成 → 综合 agent 合并输出
**提示词模板**：
创建一个 workflow，对 [目标范围] 执行扇出式处理：

扇出 [N] 个 subagent，分别处理 [模块A/模块B/模块C...] 每个 agent 独立输出结构化结果 综合 agent 合并所有结果 [可选] 输出格式：[具体要求]


### ② 对抗式验证（Adversarial Validation）
**适用**：安全修复、代码审查、高质量输出
**流程**：生成结果 → 另起 agent 以对抗姿态审查 → 不通过则返回修复 → 循环直到通过
**提示词模板**：
创建一个 workflow，对 [目标] 执行对抗式验证修复：

修复 agent → 修改代码 reviewer agent → 以对抗姿态审查，按以下 rubric： [检查项1] [检查项2] [检查项3] reviewer 必须指出至少 [N] 个问题，否则视为审查不充分 不通过则循环修复，连续通过 [N] 次才算完成


### ③ 锦标赛（Tournament）
**适用**：方案选优、架构决策、设计评审
**流程**：N 个 agent 用不同方法竞争 → 成对评判 → 淘汰制直到只剩一个
**提示词模板**：
创建一个 workflow 评估 [目标] 的 [N] 种方案：

方案A agent → 实现思路 + 代码原型 方案B agent → 实现思路 + 代码原型 方案C agent → 实现思路 + 代码原型 评判 agent → 按 rubric（[维度1/维度2/维度3]）打分 输出最佳方案 + 完整实现计划


### ④ 生成并筛选（Generate & Filter）
**适用**：创意生成、命名、方案 brainstorm
**流程**：生成多个想法 → 按 rubric 筛选 → 去重 → 保留最佳
**提示词模板**：
创建一个 workflow 生成 [目标] 的候选方案：

生成 agent → 生成 [N] 个候选方案 筛选 agent → 按 [rubric] 过滤 去重 agent → 去除相似项 输出 Top [N] 个最佳方案及理由


### ⑤ 分类并执行（Classify & Execute）
**适用**：任务路由、多类型请求处理
**流程**：classifier 判断类型 → 路由到不同 agent → 输出分类+处理结果
**提示词模板**：
创建一个 workflow 对 [输入] 执行分类路由：

classifier agent → 分析输入并分类为 [类型A/类型B/类型C] 根据分类路由到对应的处理 agent 每个类型有独立的处理逻辑 输出分类结果 + 处理结果


### ⑥ 循环至完成（Loop Until Done）
**适用**：bug 修复、错误消除、持续改进
**流程**：修复 → 验证 → 仍有问题则继续 → 直到停止条件满足
**提示词模板**：
创建一个 workflow 对 [目标] 执行循环修复：

修复 agent → 修复发现的问题 验证 agent → 检查是否仍有未修复项 如果有 → 返回 step 1；否则停止 停止条件：[具体条件，如"所有测试通过"或"无新发现"] 输出修复汇总报告


### ⑦ Prompt Chain（新增 — 元工作流）
**适用**：跨多文件大规模修改、需要严格阶段依赖、目标执行者是 Claude Code
**流程**：扇出诊断 → Quick Wins → 核心重写 → 逐阶段推进 → 对抗式审查

### 与原有六模式的区别

| 维度 | 原有六模式 | Prompt Chain 模式 |
|:-----|:----------|:-----------------|
| **执行者** | 当前上下文中的 subagent | **外部 Claude Code 实例** |
| **通信介质** | 内部 context window | **结构化提示词（可复制粘贴）** |
| **工作单元** | agent 调用 | **prompt 文件（.md）** |
| **状态传递** | 自动（共享 context） | **显式前置条件清单** |
| **失败处理** | 即时重试 | 人工介入修复后继续 |
| **Token 消耗** | 高（多个 subagent 同时运行） | **可控**（每次只执行一个 prompt） |
| **适合场景** | 单次会话内复杂任务 | **大规模、多阶段、跨会话项目** |

### 落地技巧

| 技巧 | 做法 |
|:---|:---|
| 🎯 **前置条件显式化** | 每个 prompt 开头用 ✅ 列出已完成项 |
| 📖 **"必须先读"文件列表** | 让 Claude Code 先读关键文件 |
| ✅ **可验证的 checklists** | 验证项必须是可自动化检查的 |
| 🧩 **按依赖排序** | Quick Wins 优先，核心重写次之 |
| 🚫 **不跨阶段** | 一个 prompt 只做一个阶段的事 |
| 🔄 **人工 checkpoint** | 每个 prompt 执行完检查再喂下一个 |
| 📝 **输出格式明确** | 明确要求"输出完整文件内容" |
| 🛡️ **安全审查内置** | 最终 prompt 必须有安全审查 checklist |

## 实用技巧

| 技巧 | 做法 |
|:---|:---|
| ⚡ 触发 workflow | 提示词中加 **ultracode** |
| 💾 保存 workflow | 菜单按 **s** 键 → 签入 `~/.claude/workflows/` |
| 🔁 配合 /goal | 设硬性完成要求（如"所有测试通过"） |
| 🔄 配合 /loop | 按固定间隔重复运行 |
| 💰 Token 预算 | 提示中写"只用 [N]k token" |
| 🛡️ 安全隔离 | quarantine 模式：读不可信内容的 agent 禁高权限操作 |

## 注意事项

| ⚠️ | 说明 |
|:---|:---|
| 💸 Token 消耗大 | 适合高价值复杂任务，常规编码不必硬上 |
| 📐 提示要详细 | 写得越具体效果越好 |
| 🔄 review 不是走过场 | reviewer 必须指出问题才算有效审查 |
| 🧵 Prompt Chain 需要人工介入 | 每个 phase 之间需要检查输出再继续
