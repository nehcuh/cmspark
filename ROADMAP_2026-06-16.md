## TL;DR

The audit surfaced one **critical** (privilege system is dead code) and **8 high** issues spanning security enforcement, MCP reliability, and untyped boundaries. Plan: **5 milestones over ~5-6 developer-weeks**. **Week 1** lands the test net (item 6) + 4 quick wins (items 10/11/14/16) ‚Äî these unblock everything else. **Week 2-3** resolves the privilege-system fork (item 1: recommend *delete* ‚Äî see Decision Gate) and ships evaluate hardening + cookie redaction (items 2/3/12/13). **Week 3-4** makes MCP shippable (items 5/15/17/18). **Week 5** adds defense-in-depth (items 4/7/8/9). **Deferred**: item 19 (message-router decomposition) ‚Äî only after M2-M4 stabilize. Each milestone exits on observable criteria, not "code merged".

## Decision Gates (decide BEFORE coding)

### Gate 1 ‚Äî Privilege system: wire in or delete? (item 1)

- **Recommendation: DELETE.** `privilege-manager.ts` (134 LOC), `risk-engine.ts` (298 LOC), `page-scanner.ts` (154 LOC) have zero runtime consumers; ~586 LOC of misleading dead code. Wiring it in is L effort (rebuild PrivilegeManager.getMode() + getRiskDecision() integration, fix `API_WEIGHTS` divergence, add tests for all 3 privilege tiers) for security value that overlaps heavily with items 2/12 (default-deny evaluate + navigate gate) which can ship this week with S-M effort each.
- **Alternative: WIRE IN.** Only if the team wants user-facing "readonly / standard / advanced" mode switching in the UI before end of Q3. If so, timebox the wire-in investigation to **1 day**; if it doesn't converge, fall back to delete.
- **Blast radius**: This decision **unblocks item 13** (dead-code deletion: only meaningful if we choose delete) and **shapes item 2** (evaluate hardening is partly redundant if privilege-gating covers it).

### Gate 2 ‚Äî Tool-arg validation: zod vs hand-rolled? (item 4)

- **Recommendation: ZOD.** `zod ^3.23.0` is already in `companion/package.json` but never imported. Use it. The existing `validateToolCallArguments` (`tool-definitions.ts:636`) is a partial hand-rolled validator that covers required/type/null but no semantic constraints (URL shape, range, enum). Zod schemas per tool give both, and the parse-failure ‚Üí recovery path at `adapter.ts:379-396` already exists.
- **Alternative: HYBRID.** Keep `validateToolCallArguments` as a fallback for tools without a zod schema. Marginal value, more code. Skip.

### Gate 3 ‚Äî MCP trusted-server tool aggregation (item 9)

- **Recommendation: SCAN + AUTO-AGGREGATE.** Don't require per-tool review for `trust_level:'trusted'` servers (too much UI friction). Instead, scan tool/arg descriptions with the existing injection-pattern filter before exposing to the LLM, refuse flagged tools with a clear UI warning.
- **Alternative: PER-TOOL REVIEW.** Higher safety, worse UX. Only if MCP servers will routinely come from untrusted third parties.
