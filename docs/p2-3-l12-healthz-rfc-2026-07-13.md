## 0. Grounding（已独立核验）

| 关注点 | 现状 | 位置 |
|---|---|---|
| WS server 构造 | `new WebSocketServer({ port, host:"127.0.0.1", verifyClient })`——**`ws` 库自建并持有底层 http.Server**，server.ts 无 http.Server 变量，甚至未 `import http` | `server.ts:1836-1861` |
| 端口 | `const PORT = 23401`（const）+ `config.port` 默认 23401；`const port = config.port \|\| PORT` 解析；**无 env override**；构造时同步绑定（非显式 `.listen()`），`"listening"` 事件为就绪信号 | `server.ts:31`, `config.ts:102`, `server.ts:1746,1863` |
| `startServer` | `export async function startServer(options:{onShutdown?}={})`：返 `Promise<void>`；依次 migrate/getConfig/WS 密钥/UDS lock/vision 探测/initServices/mcp.start/`new WebSocketServer`/shutdown 注册。从 `index.ts:164`（daemon）+ `index.ts:277`（plain start）调 | `server.ts:1704` |
| 既有 HTTP server | **server.ts 无**；两处独立 HTTP server 在别的端口：`settings-web.ts:220`（`/api/health`→`{status:"ok",uptime}` 先例，:335）+ `quick-action-result.ts:74`（端口 23403）。`express` 全 companion 零用 | — |
| 优雅停机（M9） | `setupGracefulShutdown` 注册唯一 handler：mcp.shutdown→historyStore.close→**`wss.close()`**→releaseLock→onShutdown。**注意**：`{port}` 构造下 `wss.close()` 会关闭 ws 自建的 http.Server；改 `{server}` 后 `wss.close()` **不**关我持有的 httpServer，需显式 `httpServer.close()` | `server.ts:2180-2212` |
| 测试模式 | **无测试调 `startServer()`**（`ws-auth-handshake.test.ts:6-12` 明述）；集成测用 bare `new WebSocketServer({port:0, host:"127.0.0.1"})` 临时端口 + `close()` 清理，复制生产逻辑 | `tests/integration/ws-*.test.ts` |

**关键结论**：要给 23401 端口加 HTTP `/healthz`，必须从 `{port}` 切到 `{server}`（显式 `http.createServer` + `httpServer.listen(port, "127.0.0.1")` + `new WebSocketServer({ server, verifyClient })`），并在 shutdown 加 `httpServer.close()`（否则 M9 停机回归——httpServer 不闭，进程不退 / 端口泄漏）。

> "healthz 端点 | L12 | WS 端口加小 HTTP `/healthz` | 0.5d"

## 2. 设计分叉（请 kimi 裁决）

### F1 — 实现方式

| 选项 | 机制 | 评价 |
|---|---|---|
| **F1-a**（推荐）切 `{server}` + 显式 http.createServer | `const httpServer = http.createServer(healthzHandler); httpServer.listen(port,"127.0.0.1"); wss = new WebSocketServer({ server: httpServer, verifyClient })`；shutdown 加 `httpServer.close()` | **ws 官方文档推荐模式**；clean、显式持有 server；handler 可独立测试。代价：改构造 + shutdown 两处 |
| F1-b 保持 `{port}`，事后 `wss.server.on("request",...)` | `wss.server`（ws 库在 `{port}` 时暴露底层 http.Server）挂 request 事件 | hacky——依赖 ws 内部 `.server` 属性（虽公开但语义不清）；request handler 在构造后挂，时序脆弱 |

**推荐 F1-a**。请裁决。

### F2 — 端点 path + 响应

| 选项 | path + body | 评价 |
|---|---|---|
| **F2-a**（推荐）`GET /healthz` → 200 `{status:"ok",uptime:process.uptime()}` | 镜像 `settings-web.ts:335` `/api/health` 先例；liveness probe 标准最小集 | **proportionate**（L12 是"小 HTTP"）；uptime 对编排（launchd/docker/supervisor）判断重启循环有用 |
| F2-b 加更多字段（version/thread 数/mcp 状态） | 丰富 | 超出 L12 范围；mcp/health 可能阻塞 probe（healthz 应恒快返） |

**推荐 F2-a**。其他 path（`/`、`/healthz/`、404）→ 404 关闭。请裁决。

### F3 — 安全（无分叉，确认）

- healthz **不经 WS 共享密钥鉴权**（liveness probe 由编排器无 WS 握手直调）。
- **仅绑 127.0.0.1**（与 WS server 同信任边界——loopback only）；远程不可达。
- `verifyClient` 只管 WS upgrade，不管 HTTP request——healthz 天然豁免。
- 响应**无敏感信息**（仅 status+uptime）。safe。
- 不记审计日志（probe 高频，会刷屏；可选 debug 级）。

### F4 — 测试

| 选项 | 机制 | 评价 |
|---|---|---|
| **F4-a**（推荐）handler 单元测 + 小集成 | 导出 `handleHealthzRequest(req,res)` 纯函数单元测（path 分流 + body 字段）；再加 1 个 http.createServer+WS({server}) 在 port 0 的集成测（http.get 打 `/healthz` 验 200+body，`/` 验 404，且 WS upgrade 仍工作） | 覆盖 handler 逻辑 + 真实 HTTP+WS 共存；遵循既有 port:0 + close() 模式 |
| F4-b 仅集成 | 只 port 0 集成 | handler 分流逻辑无独立覆盖 |

**推荐 F4-a**。请裁决。

## 3. 推荐方案（整体，待 kimi 裁决后定稿）

```
L12（companion-only）：
1. server.ts import http
2. 抽 handleHealthzRequest(req,res)：/healthz[/] → 200 {status:"ok",uptime}；else 404
3. startServer 内：
   const httpServer = http.createServer(handleHealthzRequest)
   httpServer.listen(port, "127.0.0.1")
   wss = new WebSocketServer({ server: httpServer, verifyClient })  // 删 port/host
4. shutdown（setupGracefulShutdown）：wss.close() 后加 httpServer.close()（M9 不回归）
5. tests/healthz.test.ts：handler 单元（path 分流+字段）+ port 0 集成（http.get /healthz→200、/→404、WS upgrade 仍工作）
6. 全量 npm test 绿（baseline 857）
```

## 4. 非目标

- 不加 metrics/usage HTTP 端点（P4 `/metrics 仪表盘`，remediation-plan L292）。
- 不改端口 / 不加 env override（port 仍 config）。
- 不加鉴权（loopback-only trust boundary，与 WS 同）。
- 不暴露 mcp/history 状态到 healthz（应恒快返；状态查询属 metrics）。
