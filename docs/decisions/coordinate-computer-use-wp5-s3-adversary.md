## 裁决: `SOUND WITH CAVEATS`

**FAIL 裁决与「按 plan O-2 降级实验层」的方向完全成立**：预注册门禁（plan:388「desktop 子集 ≥55%」）被实测 0/5 以 Wilson 95% 上界 43.4% 统计排除（本审重算，与评审 43.5% 一致）；合并真实 UI 组 0/9（上界 29.9%）为加强项而非依赖项；延迟双臂（fp32 2840ms / int8 3272ms）同超 plan:126 预注册的 1.5s p50 预算；退化模式经本审图像实物核验确认为模型行为而非线束缺陷。统计未被滥用——n=5 在观测 0 命中时足以排除 55% 线，「实质不可用」另有 zh 中位偏差 719px（int8.json:458，非近偏而是崩溃级）与巧合命中剔除（真实 zh ≈1/15）支撑。

**但报告的建议层（O-2 实验层规格）有两处必须修正后方可编码**：①「约束命令为英文」被本审发现处于**命令长度/句式混淆**之下——golden 集内英文 ≤21 token、中文 ≥29 token，长度与语言完全共线，报告的「模型不理解中文」归因超出数据可分性，且英文命中证据实际仅覆盖「短的直接指称命令」（较长/关系句式英文同样 0/2 失败）；②实验层建议未携带防「信任放大」的强制规格（WP4 徽标教训 + 无校准置信度 + locateAttempts 链重新引入路径）。这些不推翻 FAIL，但改变实验层要建成什么样。

**不选 VERDICT SOUND**：C-1/C-2 修正的是报告交付物中的建议部分（§5.1/§5.3），属强制细化而非可选打磨。
**不选 VERDICT REJECTED**：门禁判定本身在程序（预注册规则形式化应用）与实体（多支柱互锁、复算全中、图像实物佐证）两面均无误。

## 2. 指定攻击面逐条裁定

### C-1（MEDIUM，影响建议不影响裁决）FAIL 不过强，但「中文失效」归因处于长度/句式混淆之下，交叉分离不干净

- **一句话**：Wilson 论证程序正确（标签内 0/5 单独排除门禁线），「实质不可用」定性成立；但 golden 集**语言与 token 长度完全共线**（en ≤21 / zh ≥29，无重叠），且英文 4 case 命中模式与「短直接指称才有效」完美相关——数据同样支持「>~25 token 命令失效」假说，报告的归因（REPORT.md:42「模型不理解中文命令语义」）与实验层约束（REPORT.md:59「约束命令为英文」）都超出数据可分性。
- **证据**：① token 长度分布（§1）；② 英文内部：f-ok-en 15 tok 3.6px 中、f-icon-en 16 tok 1px 中，f-play-en 21 tok（关系句式 "in the center of the window"）149px 误中蓝图标（int8.json:127-128）、d-deskchrome-en 19 tok（"on the desktop"）815px 退化（int8.json:303-304）——**较长英文同样全灭**；③ 评审赖以分离的 f-ok-en vs f-ok-zh 对照（同图同 gt）在语言之外同时差 19 个 token（15 vs 34），分离的是 {语言, 长度, 词汇} 联合效应。
- **为何不影响 FAIL**：生产中文命令在该 tokenizer 下必然 ≥29 token 且为 OOD 词汇——两个假说（语言/长度）对「中文桌面默认兜底」同时触发，决策内容不变；zh 失效另有 fp32 臂复现（与量化无关）与中位偏差 719px 的崩溃级幅度。
- **影响在哪**：O-2 实验层的「英文约束」规格——数据实际只支持「**短的直接指称英文命令 + 干净 UI**」（n=2），较长/关系句式英文在夹具与桌面同样失效。实验层约束必须写成「英文 **且** 短命令 **且** 直接指称（非方位/关系句式）」，并以长度/句式扫描 + 校准曲线（评审 NIT-4 既有债务）测定工作包线后方可开放。
- **是否影响裁决**：否（FAIL 不动）；是（O-2 建议必须按 §4 修正）。

### C-2（LOW，强化裁决）退化点 (75,599) 是模型行为：线束 bug 类别经三层排除，本审补验最后两类

- **一句话**：英文命中排除了预处理/反归一化/会话接线/分词/命中规则五类（同图同管线）；逐案相异的退化点排除「恒定输出」类；本审目检补验「图像有效性」与「标注正确性」两类——全部排除完毕，且退化落点与论文自述的左/上位置偏置一致。
- **证据**：f-ok-en/f-icon-en 命中（int8.json:30-31/:206-207）；桌面退化点 3 个相异坐标且均为真实左缘 UI 元素（§1）；两张 eval 图有效、gt 标注与可见元素吻合（§1）；fp32 臂复现同一模式（fp32.json:83-85/:105-106/:237-238）。
- **是否影响裁决**：否——反向，把「线束 bug」替代假说基本封死，FAIL 更稳。残余理论缝：1920→768 降采样后的输入帧内容未逐帧 dump 目检（preprocessPng 输出），但「英文夹具命中 + 退化点随命令变化 + 落点为真实元素」三重证据下，该缝不具现实解释力。

### C-3（LOW，影响建议）int8 慢 15%：方向有数据、机制是推断；且「二选一」框架漏掉未评估的占优候选配置

- **一句话**：「vision 变慢、e2e int8 更慢」有分段计时硬数据（REPORT.md:20 vision 1247→1945ms@8线程；:50-51 4 核 2193→2912ms）；「conv 回退」作为机制是对分段数据的合理推断，无 ORT 算子级 profiling 佐证——报告把推断写成了事实（:20「是反优化」）。
- **更深一层（本审新增）**：`quantize_int8.py:12` 把 conv 密集的 vision_encoder 也量化了；而分段数据显示 encoder/decoder int8 反而变快（158→91 / 393→253ms）。**「vision 保持 fp32 + encoder/decoder int8」的混合变体**（体积 ≈ 366+158+~44+~137 ≈ 550-600MB，e2e @8线程理论值 ≈1247+91+253 ≈ 1.6s，可能同时优于两臂）未被评估——REPORT.md:58「要内存选 int8、要速度选 fp32」的二选一框架漏掉了潜在占优配置。
- **是否影响裁决**：否。「不得以延迟为目的选 int8」由 e2e 实测支撑，成立；混合配置属 WP5 变体决策的未尽评估项（O-2 若推进须补测）。

### C-4（MEDIUM，影响建议不影响裁决）O-2 下游：实验层建议未携带防「信任放大」强制规格，存在三条重新引入路径

- **一句话**：报告说了「默认路径不得依赖」（REPORT.md:57）但止步于此——一个「默认关闭的实验层」被用户手动开启后，UI 文案、置信度呈现与降级链排序三条路径都可能复现 WP4 徽标教训（plan:374 ⑤「徽标信任放大」）。
- **证据与路径**：① **置信度呈现**：plan:239 步骤时间线每步渲染「layer 徽标 + confidence」，而 TinyClick 无校准曲线（评审 NIT-4 未偿债务）——raw softmax 在高置信度下照样坍缩到显著点，未校准数字上屏 = 把弱证据渲染成强背书；② **开关文案**：若实验层开关不披露实测数字（zh 13.3% 含巧合、真实桌面 0/5、4 核 2.8-3.3s），中文用户开启后输入中文命令即落入 13.3% 区间——「英文约束」只写在报告里就是文档约束，必然泄漏（产品是中文桌面，用户命令天然中文）；③ **降级链重新引入**：plan:388「下载失败自动降级 L1/L3」与 O-2「低置信度要求用户框选」——实验层若进 locateAttempts 链且编排器以 raw 置信度阈值接受，显著点建议会以十字线形式呈上 L2 确认框（人审兜底仍在，但预览呈现的是带权威感的错误位置）。
- **是否影响裁决**：否。属 §4 强制规格：开关文案披露实测数字、无校准前抑制置信度显示、实验层输出永不自动被降级链接受（必经 L2 且 caption 标注「实验层建议，可能完全错误」）、包线约束代码化（非英文/超长命令/超宽输入 → 拒绝并给出原因）、显著点坍缩检测（多命令同点 → 抑制建议）。

### C-5（LOW，观察级）Case 设计噪声：命令与真实布局错配、人工标注组

- **一句话**：d-start-zh 命令「点击左下角的开始按钮」指向角落实则本机 Win11 居中任务栏、gt 在 cx=760（golden_build.py:24）——命令语义与布局错配注入噪声；桌面/设置组 gt 为人工标注（:23/:29 注释），非夹具的绘制级精度。
- **是否影响裁决**：否——方向中性偏保守（模型反正全灭，错配不构成假未中的解释力）；O-2 补测时应改为「命令语义与目标布局一致」的 case 设计。
