• The synthesis is directionally correct but uses "4 agents independently agreed" as a substitute for empirical risk analysis. Below is a Kimi+Pi adversarial review.

  **Q1 — Option A (Touch ID per call).** Thread-scoped biometric bypass is the largest security downgrade in the proposal. The Round 2 synthesis locked `host_write` behind `biometric` precisely because writes mutate host state (`computer-use-round2-synthesis.md` §4.2). If one Touch ID pre-approves all subsequent Notes creates in a thread, a prompt-injection that survives within the thread can chain destructive writes without ever touching the biometric sensor. Option A preserves non-repudiation per call. Trade-off: real fatigue after ~3 writes/60s; if fatigue metrics show drop-off, add a **time-bound session** (e.g., 5 min) rather than an unlimited thread grant.

  **Q2 — Option B (one-line notice on next call).** Silent reset relies on the user noticing the thread-switch UI, which is not always prominent. A single non-blocking line ("本线程需重新确认 Mail 读取") preserves the mental model without modal noise. Trade-off: adds a small amount of UI chatter; acceptable because it only fires on the first post-switch call.

  **Q3 — Option B (`(kind, target_app)`).** Matching by `kind` only is too coarse. `host_write` `kind: "create"` could mean Notes create-note or Finder create-folder; trusting `kind` alone lets a later call pivot to a different app within the same thread. The `target_app` is already encoded in `TargetId` per Round 2's opaque `TargetId` design (`computer-use-round2-synthesis.md` §2.1). Trade-off: requires extracting bundle id from `TargetId`, but it keeps the trust tuple honest.

  **Q4 — Option B (stay Mail-only).** Expanding `READ_ALLOWED_APPS` to Notes/Finder is scope creep that violates Round 2 §4.2's Phase 1 lock: `host_read` = Mail only, `host_write` = Notes create + Finder move. The inline checkbox for `host_read` will therefore only be meaningful for Mail in Phase 1, which is enough to validate the pattern. Trade-off: users cannot read Notes in Phase 1, but that was already the agreed scope.

  **Q5 — Option B (single-app, `string | null`).** Neither option is well-shaped for persistent cross-thread trust, but `string[]` is the worse choice. A flat array of bundle ids cannot express `(app, kind, verb)` granularity and mirrors the domain-whitelist pattern too literally. A single nullable field limits blast radius and forces Phase 2 to design a real structured schema rather than carry over a placeholder. Trade-off: power users with two trusted apps need repeated approvals.

  **Q6 — Option A (proprietary thread whitelist).** For Phase 1 thread scope, the whitelist is inherently cmSpark-specific: OS TCC has no "this thread" concept. Mirroring TCC (`Option B`) is the right long-term posture but is Phase 2 work. A "hybrid" (`Option C`) prematurely commits to two trust surfaces. Trade-off: the UI must be honest that this is a **cmSpark thread trust**, not a system setting.

  **Q7 — Option C (silent).** Thread-scoped trust lives only for the lifetime of one thread; the OS cannot realistically update AppleScript sdef mid-thread. Auto-detecting macOS version changes (`Option B`) is over-engineering for Phase 1, and `Option A`'s label adds noise for a transient trust. Trade-off: drift risk for Phase 2 persistent whitelist remains unaddressed and must be reopened then.

  **Q8 — Log all three, plus `tool_call_id`.** `original_confirmation_id` is mandatory for non-repudiation and mirrors the domain-whitelist validation pattern in `server.ts:890-985` (`handleSecurityConfirmationResponse`). `thread_id`, `bundle_id`, and `kind` define the scope and object of the grant. `tool_call_id` already exists in the surrounding log context and should be included for correlation. Trade-off: slightly larger audit records; negligible.

However, the recommendation must not hide the security cost of Q1/Q3; "4 agents agreed" is not a risk argument.

The other questions are product/engineering trade-offs; Q1 is a security boundary.
