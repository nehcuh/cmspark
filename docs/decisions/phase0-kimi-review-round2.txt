+# HostAdapter Interface — Phase 1 W4 Definition
+
+> **Date**: 2026-07-16
+> **Authority**: Round 2 synthesis §2.1 (Pi rule-of-three wins over Brief / Kimi "define first")
+> **Source material**:
+> - macOS implementation: `companion/src/host-use/darwin/` (Phase 0 spike)
+> - Linux shape: `companion/src/host-use/linux/RUNBOOK-phase0.md`
+> - Windows shape: `companion/src/host-use/win/RUNBOOK-phase0.md`
+> **Status**: design doc — implementation will follow in Phase 1 W5-W6.
+
+## Context
+
+Round 2 synthesis §2.1 explicitly defers HostAdapter interface definition to W4 (end of Phase 0 spike), AFTER three real platform implementations exist. The rule-of-three rationale: abstracting from imagined platforms gets the abstraction wrong; abstracting from three real ones gets it right.
+
+Phase 0 produced:
+- **macOS real implementation** (Mail inbox top-1 read via Swift binary + precompiled .scpt)
+- **Linux RUNBOOK** (concrete pyatspi/gdbus shape for Evolution)
+- **Windows RUNBOOK** (concrete UIAutomation shape for Outlook — expected UIAccess-blocked)
+
+All three converge on: **list targets → read one → write one**. That's the abstraction.
+
+## The interface
+
+```typescript
+// companion/src/host-use/host-adapter.ts
+
+/**
+ * Platform-agnostic host-use adapter. Each platform (darwin/linux/win) implements
+ * this against its native accessibility stack:
+ *   - darwin: AppleScript via precompiled .scpt + Swift binary (NSAppleScript)
+ *   - linux:  AT-SPI via pyatspi/gdbus shell calls + Rust tray binary
+ *   - win:    UIAutomation via PowerShell + C# WinForms tray (Phase 1.5)
+ *
+ * The interface is deliberately minimal (3 methods). Round 2 §2.1 explicitly
+ * forbids 6- or 8-method "kitchen sink" abstractions — opaque TargetId + 3
+ * methods is the rule-of-three output.
+ */
+export interface HostAdapter {
+  /** List readable targets of a given kind (inbox / calendar / notes / files). */
+  listReadTargets(kind: TargetKind): Promise<TargetId[]>
+
+  /** Read one target's current content. Returns platform-agnostic shape. */
+  readOne(targetId: TargetId): Promise<ReadResult>
+
+  /** Write to a target (create / move / update). Phase 1 limited to Notes create + Finder move. */
+  writeOne(targetId: TargetId, payload: WritePayload): Promise<WriteResult>
+}
+
+/**
+ * Opaque platform-specific target identifier. Companion code never inspects
+ * the internal structure — only passes it back to the same adapter that
+ * produced it.
+ *
+ * Platform conventions:
+ *   - darwin: `bundleId:accountId:itemId` (e.g. `com.apple.mail:INBOX:msg-12345`)
+ *   - linux:  `atspi://path/to/object` (AT-SPI registry path)
+ *   - win:    `hwnd:automationId` (UIAutomation cache key)
+ */
+export type TargetId = string & { readonly __brand: unique symbol }
+
+export type TargetKind =
+  | "mail-inbox"
+  | "mail-thread"
+  | "calendar-event"
+  | "note"
+  | "file"
+
+export interface ReadResult {
+  /** Platform-agnostic field set. Phase 0 Mail spike emits all 4. */
+  sender?: string
+  subject?: string
+  date_received?: string  // ISO 8601 (Phase 1 normalizes from AppleScript locale)
+  body_preview?: string
+  /** Future-proofing: platforms may emit additional fields. */
+  [key: string]: unknown
+}
+
+export interface WritePayload {
+  /** Kind of write — controls what `target` and `body` mean. */
+  kind: "create" | "move" | "update" | "delete"
+  /** Body content for create/update. Format depends on TargetKind. */
+  body?: string
+  /** Destination for move operations (e.g. Finder folder path). */
+  destination?: string
+}
+
+export interface WriteResult {
+  /** Opaque ID of the created/updated target, if applicable. */
+  target_id?: TargetId
+  /** Whether the write was undoable (Finder trash = yes; Mail send = no). */
+  undoable: boolean
+}
+```

+## Open design questions for review
+
+### Q1: Should `TargetId` be branded or plain string?
+
+**Current proposal**: branded for type safety (`string & { __brand }`).
+
+**Alternative**: plain `string`. Simpler, no casting needed at platform boundaries.
+
+**Recommendation**: branded. The brand is zero-runtime-cost and prevents companion code from accidentally constructing fake target ids (e.g., from LLM output without validation). Platform adapters cast at the boundary; companion code only consumes.
+
+### Q2: Is `listReadTargets` returning `[]` an error or "empty inbox"?
+
+**Current proposal**: empty array is valid (empty inbox is normal).
+
+**Implication**: companion must handle empty list gracefully — LLM should be told "no messages" rather than retry / error.
+
+**Recommendation**: keep as valid empty. Add to LLM system prompt: "if listReadTargets returns [], report 'inbox empty' to user".
+
+### Q3: Should `writeOne` accept `targetId: null` for create operations?
+
+**Current proposal**: `writeOne(targetId, payload)` where targetId may be a "parent" (e.g., Notes app for create).
+
+**Alternative**: separate `createOne(parentId, payload)` method.
+
+**Recommendation**: keep single `writeOne`. Phase 1 only does Notes create + Finder move; both can be modeled as `writeOne(parentTargetId, {kind:"create", body})`. If Phase 2 adds more write scenarios and the model strains, refactor then.
+
+### Q4: `ReadResult` extension — `[key: string]: unknown` index signature vs explicit union?
+
+**Current proposal**: index signature for forward compatibility.
+
+**Alternative**: discriminated union per TargetKind (`MailReadResult | CalendarReadResult | ...`).
+
+**Recommendation**: index signature for Phase 1 (less ceremony, platforms emit what they have). Refactor to union in Phase 2 if LLM gets confused by inconsistent shapes.
+
+### Q5: Async iterator for `listReadTargets`?
+
+When inbox has 1000+ messages, returning `TargetId[]` materializes all. Should we use `AsyncIterable<TargetId>` instead?
+
+**Recommendation**: Phase 1 keeps array. Mail inbox top-1 read is the only Phase 1 use case; large mailboxes are Phase 2+. If performance becomes an issue, add `listReadTargets(kind, {limit, cursor})` later.
+

• # Phase 0 computer-use Round 2 评审（adversarial / critical）

  范围：`e40065b..HEAD`（`b13836b` standards fixes + `9a92e85` HostAdapter interface 设计）。  
  以下按 **🔴 Critical / 🟡 Major / 🟢 已做对** 分类，带具体 `file:line` 与修复建议。

  ---

  ## 🔴 Critical

  ### 1. `host_read` 未校验 `security_token`，任意 token 都能绕过 L2 确认门
  - **位置**：`companion/src/server.ts:1124-1139`（`case "host_read"`），对照 `companion/src/server.ts:1057-1061`（`osascript_eval` 的正确做法）
  - **问题**：
    - L2 gate（`:312`）只在 `!finalParams.security_token` 时弹窗/确认；
    - `host_read` 分支拿到 `params.security_token` 后**完全不验证**，直接执行；
    - schema 里 `host_read` 的 `security_token` 又是 `optional()`（`companion/src/bridge/tool-schemas.ts:86`）。
    - 结果：调用方只要传任意非空 `security_token`（包括复用 evaluate/osascript_eval 的 token），就能跳过 `host_read` 的人工确认，直接读 Mail。
  - **建议**：在 `host_read` 分支照搬 `osascript_eval` 的 token 校验：
    ```ts
    if (params.security_token) {
      const valid = securityPolicy.validateToken(params.security_token, "host_read", "")
      if (!valid) return { success: false, error: "Invalid or expired security token" }
    }
    ```
    或者统一在 gate 处把 token 消费掉，不要传到具体 tool 分支再各自校验。

  ## 🟡 Major

  ### 2. `host_read` 在 god-mode / auto-approve 下缺少 `security.godmode_bypassed` 审计
  - **位置**：`companion/src/server.ts:428-436`（auto_approved 日志），`companion/src/server.ts:303-311`（注释提到 ADR-010）
  - **问题**：ADR-010 要求 **每次 god-mode 放行**都写 `security.godmode_bypassed`。URL gate（`:505`）做到了，但 `evaluate/osascript_eval/host_read` 的 L2 跳过只写 `security.auto_approved`，事件名不一致。`host_read` 读的是用户 Mail 这种高敏感数据，审计事件却比其他 gate 弱。
  - **建议**：当 `skipConfirmation === true` 且原因是 `allow_all_schemes` 时，对 `host_read` 也补一条 `security.godmode_bypassed`（layer: "confirmation", tool_name: "host_read"）。

  ### 3. `host_read` 的 god-mode 注释 misrepresent ADR-010
  - **位置**：`companion/src/server.ts:307-310`
  - **问题**：
    - 注释说“用户必须通过 standard phrase gate 开启 god-mode per ADR-010”，但 ADR-010 还允许直接改 `~/.cmspark-agent/config.json` 开启，**不需要短语门**；
    - 注释只提 `allow_all_schemes`，没提 `auto_approve_dangerous` 同样会跳过 `host_read` 的确认；
    - 注释只“说明”不“检查”，没有 runtime 审计/加严。
  - **建议**：改写成“`allow_all_schemes` 或 `auto_approve_dangerous` 都会跳过此门；config.json 路径无短语门；god-mode 下必须留 `godmode_bypassed` 审计”。

  ### 4. M1 evidence tag `[executed]` 被用在“用户口述观察”上，证据强度被夸大
  - **位置**：`docs/decisions/phase0-macos-gate-evidence.md:20`，`docs/decisions/phase0-go-no-go.md:16、33`
  - **问题**：你们自定义的 convention 说 `[executed] = ran the command in this session, output captured`。TCC dialog 是否出现、用户是否点击，是**用户侧 Terminal 的人工观察**，并没有被 Bash tool / 日志 capture。把它标成 `[executed]` 会让读者误以为这是可复现的自动化输出。
  - **建议**：
    - 给人眼观察的 UI 事件新增一个 tag，例如 `[reported]` 或 `[observed_by_user]`；
    - 在 TCC 名称被截图/日志 capture 之前，把“prompt named cmspark-host”从 `[assumed]` 保留，不要升级到 `[executed]`。

  ### 5. TCC attribution 从 PENDING 改成 SOFT-PASS 引入了新的“半通过”状态
  - **位置**：`docs/decisions/phase0-go-no-go.md:24-26、31-33、48`
  - **问题**：Round 2 synthesis §5.1 的 kill 信号只有 PASS/FAIL，没有 SOFT-PASS。现在文档把“名字未 capture”包装成 SOFT-PASS 并允许进入 Phase 1 W4，这相当于在没有真正验证最关键假设（TCC attribution）的情况下启动下一步。
  - **建议**：
    - 明确这是**管理上的条件前进**，不是技术 gate 通过；
    - 把“TCC 名称截图复核”从 `[ ] Optional` 改成 `[ ] Blocker for Phase 1 ship`；
    - W4 只应产出设计文档/接口定义，**不应写依赖 TCC attribution 成立的 runtime 代码**。

  ### 6. `NotImplementedOnPlatform` 类对未知平台没用上
  - **位置**：`companion/src/host-use/index.ts:19`，`companion/src/host-use/types.ts:13-18`
  - **问题**：`linux`/`win` stub 会抛 `NotImplementedOnPlatform`，但 `host-use/index.ts` 对未知平台（如 `freebsd`）抛的是普通 `Error`。既然文档说“这个类现在 load-bearing”，应该统一错误类型。
  - **建议**：`index.ts:19` 也改为 `throw new NotImplementedOnPlatform(process.platform)`。

  ### 7. HostAdapter 设计文档声称“三个真实实现”收敛，实际只有 macOS 一个能跑
  - **位置**：`docs/decisions/host-adapter-interface.md:15-20`
  - **问题**：Phase 0 只有 macOS 实装；Linux/Windows 是 RUNBOOK。说“three real platform implementations exist”是事实错误。Round 2 接受 RUNBOOK 是因为 Windows 被 UIAccess 阻塞、Linux 还没机器，但接口文档不应把 RUNBOOK 说成 implementation。
  - **建议**：改成“**one real implementation (macOS) + two RUNBOOK-shaped targets (Linux/Windows)** converge on list/read/write”；并显式列出 Windows UIAccess 阻塞尚未被验证的风险。

  ### 8. Linux RUNBOOK 的 handoff 与 interface 形状不一致
  - **位置**：`companion/src/host-use/linux/RUNBOOK-phase0.md:168-169`，`docs/decisions/host-adapter-interface.md:59`
  - **问题**：
    - RUNBOOK 说 `listReadTargets("mail")` 返回 `[{id: "atspi://..."}]`（对象数组）；
    - interface 说 `listReadTargets(kind: TargetKind): Promise<TargetId[]>`（字符串数组）；
    - RUNBOOK 用的是 `"mail"`，interface 用的是 `"mail-inbox"`。
  - **建议**：统一为 interface 的 `TargetId[]` + `"mail-inbox"`；RUNBOOK 作为 source material 必须与接口一致。

  ### 9. `TargetKind` enum 过早抽象了未经验证的 kind
  - **位置**：`docs/decisions/host-adapter-interface.md:61-67`
  - **问题**：Round 1 Pi 明确警告 early abstraction。Phase 1 实际只计划 `mail-inbox` / `note` / `file`；`mail-thread`、`calendar-event` 在三平台 spike 中都没有被验证过，尤其是 Linux Evolution / Windows Outlook 是否天然有“thread”概念还不清楚。
  - **建议**：Phase 1 把 enum 砍到 `mail-inbox | note | file`，其余 kind 在 rule-of-three 验证后再加。

  ### 10. `WritePayload` 不是 discriminated union，create/delete 语义无法用类型表达
  - **位置**：`docs/decisions/host-adapter-interface.md:78-85、147-151`
  - **问题**：
    - `kind: "create"` 时 `targetId` 到底是 parent 还是 null？文档说“parentTargetId”，但 interface 签名没体现；
    - `kind: "delete"` 时 `body`/`destination` 必须为空，但当前类型允许传；
    - `kind: "update"` 时 `body` 必填，但 interface 里 `body?: string`。
  - **建议**：改成 discriminated union：
    ```ts
    type WritePayload =
      | { kind: "create"; body: string }
      | { kind: "move"; destination: string }
      | { kind: "update"; body: string }
      | { kind: "delete" }
    ```
    并明确 `create` 的 `targetId` 语义（parent / app-scoped target / null）。

  ### 11. `ReadResult` 的 `[key: string]: unknown` 让“forward compatibility”变成 type hole
  - **位置**：`docs/decisions/host-adapter-interface.md:68-76、155-160`
  - **问题**：index signature 下，任何平台都可以往结果里塞任意字段，LLM prompt contract 会变得不稳定；而且 `ReadResult` 已知字段都是 optional，platform 可以合法返回 `{}`。
  - **建议**：Phase 1 至少去掉 index signature，严格返回 4 个字段；如果允许扩展，用 `& Record<string, unknown>` 并在 adapter 层做 schema 校验，而不是让 interface 放任。

  ### 12. `listReadTargets` 返回 array + wrapper `readOne(listReadTargets(...)[0])` 有性能/竞态问题
  - **位置**：`docs/decisions/host-adapter-interface.md:105、181`
  - **问题**：
    - 大 inbox 下 materialize 全部 `TargetId[]` 再取 `[0]`，与 Phase 0 直接读 top-1 相比性能倒退；
    - list 和 read 之间 inbox 可能变化，导致读到非目标邮件或 targetId 失效；
    - 空 inbox 时 `[0]` 是 `undefined`，直接传给 `readOne` 会崩溃。
  - **建议**：
    - 给 `listReadTargets` 加 `{ limit?: number; cursor?: string }` options；
    - Phase 1 保留一条直接的 top-1 read 路径，不要把 `hostRead(params)` 真改成 `readOne(listReadTargets("mail-inbox")[0])`；
    - 在调用处显式处理 `[]`。

  ### 13. HostAdapter interface 完全没处理 4 档确认梯度
  - **位置**：`docs/decisions/host-adapter-interface.md:127-166`，对照 `docs/decisions/computer-use-round2-synthesis.md:198-200`
  - **问题**：Round 2 §4.2 规定了 `silent / ask-once / double-confirm / biometric` 四档，Phase 1 至少要有 macOS `ask-once`(read) + `biometric`(write)，Linux 用“手动输入 6 位 nonce 不可复制粘贴”。但 interface 只定义数据操作，没说明确认发生在哪一层、adapter 是否需要暴露平台能力。
  - **建议**：在 interface 文档中新增一节“确认责任边界”：
    - companion `security-confirmation.ts` 负责按 TargetKind + WritePayload.kind 决定梯度；
    - adapter 只实现数据操作，不进确认逻辑；
    - Linux 手动 nonce 作为 platform capability 显式文档化。

  ### 14. `TargetId` branded type 的 cast 规则没文档化，存在 type hole
  - **位置**：`docs/decisions/host-adapter-interface.md:59、129-136`
  - **问题**：
    - branded type 在 TS→CommonJS 下确实是零 runtime 成本；
    - 但只要 platform adapter 或 companion 某处写 `as TargetId`，type safety 就失效；
    - 未来 `host_write` 若把 `TargetId` 暴露给 LLM，LLM 可以直接拼字符串，必须经过 runtime validation。
  - **建议**：
    - 规定“只有 adapter 内部可以把平台字符串 cast 成 `TargetId`；companion 代码禁止 `as TargetId`”；
    - 提供一个运行时 validator（例如校验 macOS `bundleId:` / Linux `atspi://` / Win `hwnd:` 前缀），并在 LLM 输入路径强制调用。
