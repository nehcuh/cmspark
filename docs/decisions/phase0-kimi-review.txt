• # Phase 0 computer-use spike — adversarial review

  > 审查范围：`companion/src/host-use/darwin/` 下的 Swift binary、构建脚本、AppleScript、TS 调用入口与黑名单。  
  > 立场：只看**致命缺陷 / 安全漏洞 / 签名-TCC-argv 边界错误**，不做 nice-to-have 点评。

  ---

  ## 先回答 6 个核心问题

  ### 1. `host.swift` 的 NSAppleScript 是否真的把 TCC attribution 绑在 `cmspark-host` 上？还是会被父进程劫持？

  **NSAppleScript 在进程内发 Apple event，TCC Automation 的归因对象确实是当前进程 `cmspark-host`**，不是 `osascript`，也不是 Node 父进程。但当前实现**没有保护好这个归因锚点**：

  - `index.ts` 通过 `CMSPARK_HOST_BIN` 环境变量或相对路径 `dist/cmspark-host` 直接 `execFile`，**没有校验签名/路径**（`host-use/darwin/index.ts:12-16,24-29`）。
  - `host.swift` 从可执行文件同级的 `host-scripts/` 加载**未签名的 `.scpt`**（`host.swift:16-25,35-40`）。

  结果：攻击者只要能替换 `.scpt` 或把 `CMSPARK_HOST_BIN` 指向另一个 ad-hoc 签名的同名 binary，就能**复用用户已经点过“允许”的 TCC 授权**跑任意 AppleScript。TCC 绑定的是 binary identity，但 binary identity 在这里没有被可信地固定。

  ### 2. codesign flag 是否足够？缺什么？

  `--sign - --options runtime` + `com.apple.security.automation.apple-events` + LV-on 是**技术上能弹出 Automation 权限对话框的最小组合**，但**不足以构成安全边界**：

  - ad-hoc 签名没有 Apple 信任的锚，任何人都能编译一个同 Bundle ID、同 entitlement 的 binary。
  - 没有资源签名：`.scpt` 不在 signature envelope 里。
  - 构建脚本只 `codesign -dv` 显示信息，**没有 `codesign --verify`** 验证签名完整性。
  - 缺少 `com.apple.security.cs.allow-dyld-environment-variables` 显式 `<false/>`（默认 false，但显式更安全）。

  Round 2 综合文档要求 companion 在 spawn 前做 `SecStaticCodeCheckValidity` 校验，当前完全没有实现。

  ### 3. `read-mail.applescript` 的 `jsonEscape` 是否健壮？

  **完全不健壮，实际上是 no-op。** 

  `read-mail.applescript:1-33` 的写法是：

  ```applescript
  set AppleScript's text item delimiters to "\\"
  set s to (text items of s) as string
  set AppleScript's text item delimiters to "\\\\"
  set s to (text items of s) as string
  ```

  这个写法**没有先把 `text items of s` 保存到变量**，而是直接 `as string`。由于当前的 delimiter 就是搜索字符串，split 完又立即用同一个 delimiter join，结果原封不动。实际测试确认：输入 `a"b` 或 `a\b` 都不会被转义。

  后果：恶意邮件里含 `"`、`\r`、`\n`、`\t` 会原样进入 JSON 字符串，导致 `JSON.parse` 失败，更严重的是**可能构造 JSON injection**（例如邮件主题里含 `","body_preview":"...` 类的片段）。

  ### 4. `index.ts` 用 `execFile` spawn `dist/cmspark-host`，路径解析是否有 TOCTOU / symlink 攻击面？

  **有。**

  - `resolveHostBinary()` 读取 `CMSPARK_HOST_BIN` 环境变量并直接返回（`host-use/darwin/index.ts:13`）。任何能污染 companion 环境变量的本地攻击者都可以把 host binary 换成自己的。
  - 即使不用环境变量，`dist/cmspark-host` 是相对 `__dirname` 的路径，安装/打包后目录权限如果不严格，可被替换或 symlink 劫持。
  - `execFile` 本身会 follow symlink，且调用前后没有签名校验。

  Round 2 文档明确要求校验 code signature 后再 spawn，这里没做。

  ### 5. LLM 能否通过 `host_read` 参数注入到 AppleScript source 里？

  **当前 params 不会进入 AppleScript source，但不是因为设计好，而是因为参数根本没被使用：**

  - `application` 只用于黑名单判断（`host-use/darwin/index.ts:19-22`），没有白名单，也没有传给 host。
  - `maxChars` 作为 argv 传给 binary（`host-use/darwin/index.ts:26`），但 `host.swift` 只读 `argv[1]`，后面的参数被直接忽略。
  - `read-mail.applescript` 是静态预编译 `.scpt`，没有插值。

  所以 **LLM 现在无法通过 params 注入 source**。但有两个隐患：

  1. 如果 `.scpt` 被替换，source 完全可控。
  2. 未来要实现 argv 注入时，**必须用 `NSAppleScript.executeAppleEvent(_:)` 传 descriptor list**，绝不能用字符串拼接把参数插进 AppleScript source。

  ### 6. vault 黑名单是否完整？

  **极不完整。** 当前只有 1Password、Keychain Access、SecurityAgent（`blacklist.ts:1-6`）。macOS 上还需要至少：

  - **浏览器**（都自带密码管理）：`com.apple.Safari`、`com.google.Chrome`、`org.mozilla.firefox`、`com.microsoft.edgemac`、`com.brave.Browser`、`company.thebrowser.Browser`（Arc）等。
  - **密码管理器**：`com.1password.1password-launcher`、`com.bitwarden.desktop`、`com.lastpass.lastpassmacdesktop`、`com.dashlane.dashlane`、`org.keepassx.keepassxc` 等。
  - **系统凭据/设置**：`com.apple.systempreferences` / `com.apple.Settings` / `com.apple.keychainaccess` / `com.apple.Passbook`（Wallet）/ `com.apple.authenticatorcored` 等。
  - **银行/金融 App**：无法穷举，应加 heuristics。
  - **Terminal / SSH 客户端**：`com.apple.Terminal`、`com.googlecode.iterm2` 等（滚动缓冲区里可能有密钥/密码）。
  - **硬件钱包 / 加密钱包**：Ledger Live、Electrum 等。

  更关键的是，**纯 bundle-id 精确匹配容易被绕过**，应配合 `AXSecureTextField` 角色、窗口标题 heuristics（`password/login/bank/2FA`）等兜底。

  ## 一句话结论

  Phase 0 的 TCC 锚点设计方向是对的（ad-hoc signed Swift binary + NSAppleScript in-process），但**实现上把锚点暴露了**：`.scpt` 不签名、companion 不校验 host binary、AppleScript JSON 转义完全失效。这三条任意一条都能在用户点过一次“允许”后，把 `host_read` 变成任意 AppleScript 执行器。修复前不应进入 Phase 0 用户测试。
