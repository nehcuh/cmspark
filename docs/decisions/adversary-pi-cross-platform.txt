Pi-sub Round 2 评审。实现成本视角，不和 Kimi 重复安全/产品视角。先说结论：Q5 我支持 Option A（同步 ship），但条件是 Q6 interface 必须**先 darwin 跑通再定义**，Kimi 的「Option A + interface 前置」是错的。

## Q6 HostAdapter interface — 反对 Kimi，反对前置

**Kimi 错在哪**：Kimi 说「Option B 会让 interface 设计被 macOS 实现绑架」，但 Kimi 没写过跨平台 native adapter。真实情况是：**前置定义的 interface 会被你对 3 个平台的想象绑架**，而不是被 darwin 实现绑架。后者至少基于真实跑通的代码。

具体讲：你今天想象 HostAdapter 该有 `runScript(template, args)` 这种方法。Windows 上等价的是 PowerShell ScriptBlock + ConstrainedLanguage，AT-SPI 上根本没有「script」概念，是 AT-SPI role + action 调用。你今天定义的 `runScript` 在 Linux 上是 vacuous——你硬塞进去一个 `runAction(role, action)` 也对，但 darwin 上的 `applescript_run` 是 AppleScript 独有的「自然语言脚本」，根本不是 action。

**Pi 的方案（brief Option B 改良版）**：

1. Phase 0（W1-W3）：3 平台并行 spike，**各自裸写**，darwin 走 Round 1 的 `applescript_run` + 预编译 .scpt，Win 写最小 UIAutomation PoC（读 Outlook 收件箱 top 1），Linux 写 AT-SPI 最小 PoC（读 Evolution mail top 1）。
2. Phase 1（W4）开始前**一次性定义 interface**，基于 3 套真实跑通的实现做 abstract data type，而不是想象中的 type。
3. interface 粒度：**只抽象 3 个方法**——`listReadTargets()`, `readOne(targetId)`, `writeOne(targetId, payload)`。其他全平台特异（bundle id / hwnd / window class）走 opaque token，server.ts 不关心 token 内部结构。

不要 abstract 6 个、8 个方法。3 个就够 Phase 1 验证。

## Q7 native vs Node 跨平台库 — 强烈反对 Node 库

**Option B / Option C（Node 库 nut.js/robotjs）是陷阱**。理由：

1. **nut.js / robotjs 是 GUI 自动化库，不是 host-use 库**。它们能点鼠标、敲键盘、读屏幕像素，**不能**「读 Outlook 第一封邮件的正文」——因为那要走 UIAutomation tree 而不是 mouse/keyboard。用 nut.js 实现「读邮件」你得模拟点 Outlook → 等窗口 → 读 label 文本，这比直接调 `UIAutomation::GetRootElement::FindFirst(Mail → message)` 慢 10-100 倍且 brittle 100 倍（窗口布局一变就挂）。
2. **Win32 UIAutomation API 延迟实测**：`IUIAutomationElement::FindFirst` 走 cache request 大约 5-30ms，不缓存走 100-500ms。Outlook 这种重 Electron/Win32 混合 app，UIAutomation tree 遍历可能 1-3s。这对 LLM tool 往返是可接受的（LLM 一次调用本来就 1-3s），但你必须用 cache request + TreeScope，nut.js 这种「截图找按钮」做不到。
3. **AT-SPI 事件循环成本**：`atspi_event_listener_new` 一旦注册，GLib main loop 会持续 poll `/org/a11y/atspi/accessible/<n>` D-Bus 路径。Electron 应用（Code、Slack）的 AT-SPI tree 有 5000+ 节点，全量 walk 一次 5-15s。你必须用 `getRelationSet` + lazy traverse，不是 BFS。Node 没有现成的 AT-SPI binding（`node-atspi` 这种 lib 几乎无维护），你最终还是得绑 `libatk-bridge-2.0.so` 自己写 N-API addon。
4. **Node native addon 构建复杂度**：`node-gyp` 在 Win 上要 Visual Studio Build Tools（5GB+ 下载），Linux 上要 python + make + g++，macOS 上要 Xcode CLT。**用户装 cmSpark 时不会装这套**。你发布预编译 binary 时要为 (3 平台 × Node ABI 版本 × 架构) 组合预编译——Node 20 / 22 / 23 × darwin-arm64 / darwin-x64 / linux-x64 / linux-arm64 / win-x64 = 12+ prebuilds。nut.js 自己有 prebuilds，但你的 N-API addon 没有。
5. **Node 跨平台 GUI 库的维护负担**：nut.js 3 人维护，robotjs 主要维护者 1 人。这些库对 macOS Sonoma 的 `CGEventTap` 权限变化、Windows 11 的 UIAccess 新限制、Wayland portal 权限——**响应都慢半拍**。Round 1 共识「tray 必须独立 binary 不和 companion 同进程」是因为同进程被一锅端；用 Node 跨平台库等于把 GUI 操作拉回 companion 进程，Round 1 的 N3（双信道）整个作废。

**Pi 的方案**：Option A（native 优先），但分梯度：

- **macOS**：保留 Round 1 资产（`osascript_eval` + 预编译 `.scpt` + Swift tray），**不重写**。
- **Windows**：UIAutomation 走 PowerShell ScriptBlock + `ConstrainedLanguage` 模式（不需要 C++ binary，PowerShell 自己 host CLR）。tray 走 C# WinForms `NotifyIcon`（~200 行，独立 exe，Authenticode 签名）。
- **Linux**：AT-SPI 走 `gdbus call org.a11y.atspi.Registry`（shell 调用，不要 N-API addon），tray 走 Rust + `ksni` 或 `tray-icon` crate（独立 binary，~300 行 Rust）。

三套实现 3 套语言（TypeScript / C# / Rust），看起来浪费，实际上每套都是该平台**最少代码量**跑通 host-use 的路径。Node 跨平台库是「单语言覆盖三平台」的幻觉，代价是每平台都做得不好。

## Q9 IPC 抽象 — 部分同意 Claude 主线

同意「daemon IPC 抽象、tray 不抽象」的结论，但实现细节要补：

1. **UDS / Named pipe 用 Node `net` module 抽象是对的**。Windows 上 `net.createServer({ path: '\\\\.\\pipe\\cmspark-agent' })` 直接 work。但 ACL 要在 Windows 上显式 set——`ConvertStringSecurityDescriptorToSecurityDescriptorW` 配 `D:P(A;;FA;;;SY)(A;;FA;;;BA)(A;;FA;;;OW)`，否则 named pipe 默认 Everyone 可写，和 UDS 0600 不等价。`net` module **不**自动加 ACL，你必须自己调 `fs.chmod` 不行（named pipe 没 POSIX mode），得调 N-API 或 PowerShell 设置。
2. **不要抽象 tray**。Round 1 共识「tray 必须 Swift 独立 binary」是基于「macOS 同进程被一锅端」。Windows tray（C# NotifyIcon）和 Linux tray（Rust ksni）各自独立 binary，复用度几乎为 0。强抽 interface 等于在三套独立 binary 上加一层毫无意义的 type 层。
3. **抽象层只抽象「确认消息协议」**——server.ts → tray binary 的 JSON-RPC payload。三平台 tray binary 都实现同一套 `{cmd:"confirm", nonce, summary, risk_level}` → `{response:"allow"|"deny"}` 协议，传输层（stdin / named pipe / UDS）各自实现。这才是值得抽象的层。

## Kimi 漏掉的风险

1. **AppleScript sdef 不稳定（macOS）**：Mail.app 的 AppleScript dictionary 在 macOS 14.x → 15.x 之间变过 2 次（`message` 对象的 `mailbox` 属性 vs `account` 属性）。Phase 1 预编译 `.scpt` 模板**绑定 OS 版本**，用户从 Sonoma 升 Sequoia 后 .scpt 可能 runtime error。Win UIAutomation 没这问题（COM contract 稳定），Linux AT-SPI 介于两者之间。Phase 1 必须有「.scpt 失败时降级到 osacompile runtime」的 escape hatch，但 escape hatch 本身是 Round 1 Pi 反对的（osacompile 300-500ms + Unicode bug）。**实际折中**：bundle 多版本 .scpt，按 `ProcessInfo.operatingSystemVersion` 选——又一项 macOS 独有的维护成本。

2. **Windows UIAccess 标志**：Phase 1 想用 `SendInput` 模拟点击（即使只是 Outlook UI 树读取，未来扩展也要）时，未签 binary 的 UIAccess 申请会被拒。Win11 24H2 之后，未签名 binary 调 `SetForegroundWindow` 也会被 silently ignore（UIPI）。这意味着 Windows host-use 工具的「前景窗口操作」能力**默认不可用**，必须先过 Authenticode 签名 + UIAccess manifest。这和 Kimi 提的 SmartScreen 是**两个独立阻塞**，Kimi 只说了 SmartScreen。

3. **AT-SPI 在 Electron 应用上的死锁**：Chrome / VSCode / Slack 等 Electron 应用的 AT-SPI tree 由 Chromium 的 `AccessibilityObject` 暴露，但 main loop 一旦 busy（比如 Slack 收消息）整个 tree 停止响应 D-Bus query。`gdbus call --timeout 5000` 经常超时返回空。Phase 1 Linux 测试如果选 Evolution（原生 GTK）能跑通，选 Slack 就挂——会让团队误以为「Linux host-use 不行」，其实只是 AT-SPI + Electron 兼容性差。Phase 0 Linux spike 必须明确「只支持 GTK / Qt 原生 app」，Electron app 列入 Phase 2+。

4. **跨平台 trust root 校验的成本**：Round 1 共识 macOS Swift tray 用 ad-hoc + `SecStaticCodeCheckValidity`。Windows 上等价的是 `WinVerifyTrust` + `CERT_E_CHAIN` 处理——但 Authenticode 链验证冷启 500-2000ms，每次 companion 启动都跑会拖慢。Linux 上 GPG 验证基本没人用，AppImage 内置签名机制（AppImageLauncher）覆盖率 <5%。**实际可行方案**：Linux 干脆不校验 binary 完整性，靠 file system permission（`chmod 500`，root owned）做 trust root。这违反 Round 1 双信道假设，但 Linux 桌面用户本来就接受这种 model。

## 对 Kimi「砍一个平台」的立场

**部分同意**。从实现成本角度：

- **同意砍 Windows**：如果没有 EV cert + 公司主体，Windows Phase 1 投入产出比 < 0.3。但**不是因为 Kimi 说的 SmartScreen**——更深层原因是 **UIAccess + SendInput + 前景窗口操作这一整套 native automation 在未签名 binary 上根本不可用**，没签 cert 你连「读 Outlook 收件箱」这种 read-only 都跑不通（Outlook 是 UIAccess-protected 进程，未签 binary 的 UIAutomation client 看不见完整 tree）。Kimi 漏了这层。

- **反对砍 Linux**：Kimi 说 Linux 安全承诺最难兑现，技术上对，但 Linux 用户基数小（<10% 桌面份额），**风险敞口也小**。Linux 实现成本最低（AT-SPI + bash，无 binary 签名成本，无公司主体要求），适合作为 Phase 1 的「第二平台」验证跨平台架构。砍 Linux 等于放弃验证 HostAdapter interface 的跨平台性。

## 最终 verdict

**Option A（同步 ship macOS + Linux）+ Windows 推 Phase 1.5 等 EV cert**。理由：

1. 同步 ship macOS + Linux 验证 HostAdapter interface 的跨平台性（接口设计基于 2 套真实跑通的实现，不是想象）。
2. Windows 单独推后，因为：未签名 binary UIAccess 不可用 + SmartScreen 拦截 + 公司主体要求三重阻塞。这三条任一不解决 Windows 都不该进场。Kimi 只看到 SmartScreen，漏了 UIAccess。
3. 工期：macOS + Linux 同步 12-14 周（Linux 比 macOS 多 2 周，AT-SPI 兼容性测试 + Rust tray binary 学习曲线）。Windows 等 EV cert 落地后单独 6-8 周。总工期 18-22 周，比 brief 的 Option A（12-16 周三平台）多 4-6 周，但**省下来的 4-6 周避免了 Windows 未签名 binary 在用户机器上 70%+ 装不上的灾难**。

如果用户坚持三平台同步 ship，**前置条件是先批准 Windows EV cert 预算（$499/yr + 公司主体注册）**。否则「Windows Phase 1」只是承诺，不是工程。
