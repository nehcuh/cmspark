# ADR-005: Cookie 信任域安全模型

**日期**: 2026-05-24 | **状态**: 已确认

## 背景

浏览器 Agent 需要读取和管理 Cookie 以支持认证状态保持（如 SSO 自动复用），但任意访问 Cookie 是严重的安全风险。需要在可用性和安全性之间取得平衡。

## 决策

采用 **信任域白名单** 模型：

- 用户配置信任域列表（如 `*.company.com`、`example.com`、`*`）
- 信任域外的 Cookie 操作被阻断
- 通配符支持：`*.company.com` 匹配 `hr.company.com`、`finance.company.com`
- `*` 全局通配符仅用于开发/测试环境

错误分类：Cookie 未授权访问归类为 `security` 级别，不可重试。

## 权衡

### 优势

- **白名单模型更安全**：默认拒绝，显式授权
- **通配符支持**：一个配置覆盖企业内部所有子域
- **与 SSO 场景契合**：企业通常有集中的认证域（`*.company.com`）

### 劣势

- **配置负担**：用户需要预先知道要访问哪些域
- **`*` 太危险**：全局信任域一个配置失误即暴露全部 cookie
- **无上下文感知**：无法区分"读取 cookie 做认证"和"窃取 cookie 发送到外部"

## 替代方案

**每次确认模式**：每次 Cookie 操作弹出确认。被否决，因为用户体验差，且频繁确认会产生"确认疲劳"导致用户盲目点击。

**域名黑名单**：默认允许，列出禁止域。被否决，因为遗漏一个域风险太大。

## 后果

- 每个 Cookie tool（`get_cookies`、`set_cookie`、`delete_cookie`、`list_all_cookies`）在执行前都会经过信任域检查
- `list_all_cookies` 全局操作仅在 `*` 存在时放行
- 未来如果要支持"运行时动态确认"（如检测到新的 SSO 域自动建议加入信任列表），需要改造 SecurityConfirmationManager
