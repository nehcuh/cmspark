# ADR-003: SQLite 操作历史存储

**日期**: 2026-05-24 | **状态**: 已确认

## 背景

CMspark 需要记录每次 tool call 的执行详情（工具名、参数、结果、耗时、时间戳），支持按线程分组、全文搜索、JSON 导出、可配置保留天数。

## 决策

采用 **SQLite (better-sqlite3)** 作为操作历史存储引擎，数据文件位于 `~/.cmspark-agent/history.db`。

## 权衡

### 优势

- **零配置**：无需安装数据库服务，文件即数据库
- **查询能力**：SQL 支持复杂过滤、聚合、按时间/线程/工具名查询
- **性能**：better-sqlite3 是同步 API，高于异步包装器
- **可移植**：单文件，备份和迁移简单

### 劣势

- **同步阻塞**：写入操作会阻塞 Node.js 事件循环（当前操作量级可忽略）
- **无复制**：单机部署，不支持多进程共享
- **Schema 变更**：需要手动迁移，无自动化

## 替代方案

**JSON 文件**：简单但查询能力弱，大文件读写性能差。被否决。

**PostgreSQL/Supabase**：远程数据库方案，适合多机同步，但当前 MVP 阶段过度设计。

**SQLite + WAL 模式**：当前未启用 WAL（Write-Ahead Logging）。如果未来出现写冲突，启用 WAL 是低成本改进。

## 后果

- `better-sqlite3` 需要 native 编译，可能在特定 Node.js 版本或平台（ARM）出现问题
- 保留策略（`history_retention_days`）需要定时清理任务（当前未实现）
