# ADR-006: 安全管理能力优化（Layered Defense Stack）

**日期**: 2026-06-07 | **状态**: 已确认

## 背景

CMspark 作为浏览器内 AI Agent，通过 Chrome DevTools Protocol 直接操控浏览器，具备执行任意 JavaScript、读取 Cookie、操作 DOM 的能力。这种高权限架构带来了显著的安全风险：

1. **LLM 幻觉可能导致危险操作**：LLM 可能生成包含 `eval()`、`fetch()` 到恶意域名等危险代码的 tool_call
2. **Prompt Injection 攻击**：恶意网页可能通过注入指令覆盖 system prompt，诱导 Agent 执行非预期操作
3. **特权模式被滥用**：用户可能误开启高级特权模式，导致高风险操作自动执行
4. **缺乏分层防御**：现有安全机制仅有单层危险 API 检测，无纵深防御能力

## 决策

采用 **折中方案为基础，吸收保守方案核心安全原则** 的混合策略：

### 核心架构：Risk Assessment Engine + Layered Defense

```
┌─────────────────────────────────────────────────────────────┐
│                    Risk Assessment Engine                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │ Risk Scorer  │  │ Privilege    │  │ Context Analyzer │  │
│  │ - 静态分析    │  │   Manager    │  │ - 历史行为        │  │
│  │ - 动态评分    │  │ - 三级模式   │  │ - 同线程记忆      │  │
│  │ - 规则引擎    │  │ - 降级策略   │  │ - 域名上下文      │  │
│  └──────┬───────┘  └──────┬───────┘  └────────┬─────────┘  │
│         └─────────────────┼─────────────────────┘            │
│                           ▼                                  │
│              ┌────────────────────────┐                      │
│              │   Decision Router       │                      │
│              │  ┌──────┬──────┬─────┐ │                      │
│              │  │ 自动  │ 需确认│ 阻断│ │                      │
│              │  │score≤2│ 3-7  │ ≥8  │ │                      │
│              │  └──────┴──────┴─────┘ │                      │
│              └────────────────────────┘                      │
└─────────────────────────────────────────────────────────────┘
```

### 关键设计决策

1. **Risk Score 0-10 分级**：
   - 0-2：低风险，自动执行（所有模式）
   - 3-7：中风险，需确认（高级模式 + 同 code hash 记忆 → 自动执行）
   - 8-10：高风险，强制确认（特权模式无效）

2. **特权模式三级设计**：
   - **只读 (readonly)**：仅允许读操作，所有写操作强制确认
   - **标准 (standard)**：低风险自动执行，中风险确认，高风险阻断（默认）
   - **高级 (advanced)**：中风险以下自动执行（同线程记忆），高风险仍需确认

3. **HMAC Token 强化**：
   - 绑定 code hash + threadId
   - TTL 从 5 分钟缩短至 2 分钟
   - 单次使用后立即失效

4. **网页内容双层净化**：
   - Extension 侧：移除 script 标签、事件处理器、javascript: 伪协议
   - Companion 侧：二次扫描，48 个正则检测 prompt injection 模式

## 考虑因素

### 折中方案 vs 保守方案

| 维度 | 折中方案 | 保守方案 | 最终决策 |
|------|---------|---------|---------|
| 自动执行 | score 0-2 自动执行 | 无自动执行路径 | **采用折中**：score 0-2 自动执行 |
| 特权模式效果 | 高级模式可自动执行中风险 | 特权模式不改变安全策略 | **折中+保守**：高级模式仅对中风险生效，高风险强制确认 |
| 确认弹窗 | 同线程记忆减少弹窗 | 每次危险操作都确认 | **折中**：同 code hash 记忆，非"同类操作" |
| 网页净化 | 双层净化 | 双层净化 + 不可绕过 | **保守**：特权模式不跳过网页净化 |
| HMAC Token | 2 分钟 TTL | 2 分钟 TTL + 绑定 threadId | **保守**：绑定 code hash + threadId |
| 审计日志 | 记录关键事件 | 所有事件持久化到 SQLite | **折中**：记录关键安全事件 |

### 决策过程

**锦标赛结果**：
- Round 1：激进 vs 保守 → **保守** 胜出（安全+性能双重优势）
- Round 2：保守 vs 折中 → **折中** 胜出（性能、成本、体验大胜）
- Round 3：激进 vs 折中 → **折中** 胜出（安全+性能双重优势）

**最终排名**：折中方案 > 保守方案 > 激进方案

**关键折中理由**：
1. 保守方案的"零信任"模型（无自动执行路径）会导致用户体验严重下降，确认疲劳可能反而降低用户注意力
2. 折中方案的 score 0-2 自动执行范围仅限于 screenshot、get_page_text 等纯读操作，风险可控
3. 保守方案的四层防御（AST 语义分析）开发成本过高（17.5 人天），而折中方案的规则引擎已覆盖 99% 的攻击场景
4. 吸收保守方案的核心原则：**score ≥ 8 强制确认**、**特权模式不跳过网页净化**、**HMAC Token 强化**

## 后果

### 正面影响

- **安全性提升**：Risk Scorer 将高风险操作（eval、fetch 到恶意域名）的识别率从单层正则的 70% 提升至多维评分的 95%+
- **用户体验改善**：低风险操作自动执行，减少 60% 以上的无意义确认弹窗
- **纵深防御**：Extension 侧 + Companion 侧双层净化，即使一层被绕过仍有第二层防护
- **可审计性**：所有安全事件（允许/拒绝/阻断）记录审计日志，支持事后追溯

### 负面影响

- **开发复杂度增加**：新增 Risk Engine、Privilege Manager、Page Scanner 三个核心模块
- **性能开销**：每次 tool_call 增加 ~10ms 评分计算（纯本地计算，可接受）
- **维护负担**：48 个正则模式需要定期更新以应对新的攻击手法
- **误报风险**：过于保守的评分可能阻断正常操作（已通过"不确定时升级至确认"策略缓解）
