# CMspark 菜单栏服务架构性能审查报告

> 审查日期: 2026-06-07
> 审查维度: 性能（内存、CPU、磁盘、启动时间、电池/能耗、系统启动影响）
> 审查范围: 方案A（激进/SwiftUI）、方案B（折中/Electron）、方案C（保守/launchd）

---

## 一、总体结论

| 维度 | 方案A（激进） | 方案B（折中） | 方案C（保守） |
|------|-------------|-------------|-------------|
| **综合性能评分** | **8.5 / 10** | **5.5 / 10** | **7.0 / 10** |
| 常驻内存 | 最优 | 最差 | 中等 |
| 启动延迟 | 中等 | 中等 | 最优 |
| CPU 开销 | 低 | 中 | 中 |
| 磁盘占用 | 中等 | 最大 | 最小 |
| 电池/能耗 | 最优 | 最差 | 中等 |
| 系统启动影响 | 低 | 高 | 低 |

**结论**：方案A（SwiftUI + pkg）在性能维度全面领先，方案C（launchd）性价比最高，方案B（Electron）在性能上最不理想但开发效率最高。

## 二、方案A：激进方案（SwiftUI 原生菜单栏 + pkg 打包）

### 2.1 性能评分：8.5 / 10

### 2.2 资源占用分析

| 资源 | 占用量 | 评价 |
|------|--------|------|
| **菜单栏应用内存** | ~20-40MB（SwiftUI 原生） | 优秀。SwiftUI 无渲染引擎开销，纯 AppKit/NSStatusBar |
| **Companion 内存** | ~50-100MB（Node.js + pkg 运行时） | 良好。与现有 CLI 模式一致，无额外开销 |
| **总常驻内存** | ~70-140MB | 优秀。三个方案中最低 |
| **CPU 空闲占用** | <1% | 优秀。Swift 进程几乎零开销，Companion 仅 WS 保活 |
| **安装包体积** | ~80-100MB（.dmg，含 pkg 二进制） | 中等。pkg 含完整 Node.js 运行时，体积偏大 |
| **启动时间** | 1-3 秒（点击到 Companion 就绪） | 良好。pkg 二进制启动略慢于原生 Node，但可接受 |
| **系统启动影响** | 低（SMAppService 延迟加载） | 优秀。macOS 13+ 推荐方式，不阻塞登录 |

### 2.3 性能瓶颈识别

| # | 瓶颈 | 严重程度 | 说明 |
|---|------|----------|------|
| 1 | **pkg 打包二进制启动延迟** | 中 | pkg 将 Node.js 运行时与源码打包为单文件，启动时需解压/加载 ~80MB 二进制，冷启动比原生 Node 慢 0.5-1 秒 |
| 2 | **Companion 内存基线高** | 中 | Node.js 进程本身 V8 堆 + 依赖加载后常驻 50-100MB，这是 Node 生态固有限制，与方案无关 |
| 3 | **日志文件膨胀** | 低 | 长期运行下 `~/.cmspark-agent/logs/` 可能积累数 GB；Swift 侧和 Companion 侧双写日志 |
| 4 | **pkg 体积膨胀** | 低 | 含完整 Node.js 运行时 + 所有 npm 依赖，无法 tree-shake；后续功能增加体积线性增长 |

### 2.4 优化建议

| 瓶颈 | 优化建议 | 预期收益 |
|------|----------|----------|
| pkg 启动延迟 | 1. 使用 `pkg --compress Brotli` 减小体积同时加速解压；2. 调研 Node 20+ SEA（Single Executable Apps）替代 pkg，启动更快；3. 菜单栏启动时预加载 Companion 进程（用户点击前已在后台准备） | 启动时间从 3s 降至 1-1.5s |
| Companion 内存 | 1. 启动时设置 `--max-old-space-size=128` 限制 V8 堆；2. 延迟加载非核心模块（如 skill 引擎按需初始化）；3. 使用 `worker_threads` 隔离 LLM 适配器，主进程保持轻量 | 内存降低 20-30% |
| 日志膨胀 | 1. Swift 侧实现日志自动清理（保留 7 天，上限 100MB）；2. Companion 侧增加日志压缩归档；3. 提供"一键清理日志"菜单项 | 避免磁盘无限增长 |
| 体积膨胀 | 1. 使用 `sea`（Node 20+）替代 pkg，仅打包必要文件；2. 构建时启用 `ncc` 或 `esbuild` 做 dead code elimination；3. 将内置技能从 binary 中剥离，首次运行时下载 | 体积从 100MB 降至 40-60MB |

## 三、方案B：折中方案（Electron 主进程-only）

### 3.1 性能评分：5.5 / 10

### 3.2 资源占用分析

| 资源 | 占用量 | 评价 |
|------|--------|------|
| **Electron 主进程内存** | ~80-150MB | 差。即使无 Renderer Window，Electron 仍需加载 Chromium 内核基础库 |
| **Companion 内存** | ~50-100MB | 良好。与现有模式一致 |
| **总常驻内存** | **~130-250MB** | **差。三个方案中最高，接近方案A的 2 倍** |
| **CPU 空闲占用** | 1-3% | 中等。Electron 主进程有事件循环和 V8 GC 开销 |
| **安装包体积** | ~150-200MB（.app，含 Electron 框架） | 差。Electron 框架本身 ~120MB |
| **启动时间** | 2-4 秒（Electron 初始化 + Companion spawn） | 中等。Electron 主进程启动即需 1-2 秒 |
| **系统启动影响** | 高（Electron 框架加载阻塞） | 差。开机自启时拖慢登录速度 |

### 3.3 性能瓶颈识别

| # | 瓶颈 | 严重程度 | 说明 |
|---|------|----------|------|
| 1 | **Electron 主进程内存过重** | **高** | 即使"主进程-only"，Electron 仍需加载 Chromium Content Module、V8、Blink 等核心库，内存基线 80-150MB，远超菜单栏应用合理范围 |
| 2 | **双 Node 运行时共存** | 高 | Electron 内置 Node.js + Companion 独立 Node.js 进程，两套 V8 堆、两套事件循环、重复加载相同 npm 依赖 |
| 3 | **开机自启拖慢系统登录** | 中 | Electron 应用加入 Login Items 后，系统启动时需加载完整框架，显著增加登录时间 |
| 4 | **磁盘空间占用大** | 中 | `.app` 体积 150-200MB，是方案A的 2 倍、方案C的 N 倍；自动更新时旧版本残留进一步膨胀 |

### 3.4 优化建议

| 瓶颈 | 优化建议 | 预期收益 |
|------|----------|----------|
| Electron 内存过重 | 1. **强烈建议迁移到 Tauri**（Rust + WebView，内存 ~30-50MB）；2. 若坚持 Electron，使用 `--disable-gpu`、`--disable-software-rasterizer` 减少 Chromium 初始化；3. 设置 `app.disableHardwareAcceleration()` | 内存降低 30-50%（但仍远高于 Swift） |
| 双 Node 运行时 | 1. 将 Companion 作为 Electron 的 `BrowserWindow` 内嵌（不推荐，破坏架构）；2. 使用 Electron 内置 Node 直接运行 Companion 代码（需大量改造）；3. **最佳方案：放弃 Electron，改用方案A或C** | 消除重复运行时 |
| 开机自启拖慢 | 1. 延迟 10 秒启动 Companion（菜单栏先起，后台再起）；2. 提供"启动时不自动启动 Companion"选项；3. 使用 `node-auto-launch` 的隐藏启动模式 | 减少登录阻塞感 |
| 磁盘占用 | 1. electron-builder 启用 `asar` 压缩；2. 自动更新时清理旧版本；3. 剥离非必要资源（如多语言包） | 体积降低 20-30% |

## 四、方案C：保守方案（launchd + node-notifier）

### 4.1 性能评分：7.0 / 10

### 4.2 资源占用分析

| 资源 | 占用量 | 评价 |
|------|--------|------|
| **菜单栏代理内存** | ~40-80MB（Node.js + node-notifier） | 中等。node-notifier 依赖 terminal-notifier 二进制，但本身仍是 Node 进程 |
| **Companion 内存** | ~50-100MB | 良好。与现有模式一致 |
| **总常驻内存** | ~90-180MB | 中等。低于方案B，高于方案A |
| **CPU 空闲占用** | <1%（无轮询时）/ 1-2%（轮询时） | 中等。取决于状态检测机制 |
| **安装包体积** | ~0MB（依赖现有 npm 安装） | 最优。不引入新框架，仅增加脚本和 plist |
| **启动时间** | 0.5-1 秒（launchd 已预加载 Companion） | 最优。launchd 在系统启动时已完成加载 |
| **系统启动影响** | 低（launchd 并行调度） | 优秀。系统原生守护进程管理，不阻塞用户登录 |

### 4.3 性能瓶颈识别

| # | 瓶颈 | 严重程度 | 说明 |
|---|------|----------|------|
| 1 | **双 Node 进程内存叠加** | 中 | 菜单栏代理（Node.js）+ Companion（Node.js）= 两套 V8 堆，总内存 90-180MB；node-notifier 本身也依赖 terminal-notifier 子进程 |
| 2 | **WebSocket 状态轮询开销** | 中 | 文档提到"每秒轮询 WebSocket 状态"，即使改为 5 秒，仍存在不必要的 CPU/网络开销；空闲时持续消耗资源 |
| 3 | **terminal-notifier 子进程开销** | 低 | node-notifier 每次显示通知或更新菜单状态时可能 spawn terminal-notifier 进程，产生短暂 CPU 峰值 |
| 4 | **launchd 频繁重启风暴** | 低 | `KeepAlive=true` 下若 Companion 持续崩溃，launchd 可能高频重启（文档提到需配置 `ThrottleInterval`） |

### 4.4 优化建议

| 瓶颈 | 优化建议 | 预期收益 |
|------|----------|----------|
| 双 Node 进程 | 1. **将菜单栏代理合并到 Companion 进程内**：Companion 启动时自行调用 node-notifier 创建菜单栏，避免双进程；2. 使用 `cluster` 模块共享内存；3. 菜单栏代理改用纯 AppleScript（零 Node 开销） | 内存降低 40-60MB |
| WS 轮询开销 | 1. **改用 Unix Domain Socket 或文件系统事件**：监听 PID 文件 mtime 或 UDS 状态文件，零开销；2. 使用 `fsevents` 监听日志文件变化推断状态；3. Companion 主动写入状态文件，菜单栏被动监听 | CPU 空闲占用降至 ~0% |
| terminal-notifier | 1. 使用 macOS 原生 `osascript` 替代 node-notifier，避免额外依赖；2. 菜单栏图标状态更新改为事件驱动而非轮询驱动 | 消除子进程 spawn 开销 |
| launchd 重启风暴 | 1. 严格配置 `ThrottleInterval=30` 和 `LaunchLimitInterval=300`；2. 连续崩溃 3 次后写入标记文件暂停重启；3. 菜单栏代理检测崩溃频率并通知用户 | 避免系统资源被耗尽 |

## 五、方案间性能对比

### 5.1 综合性能矩阵

| 性能维度 | 权重 | 方案A | 方案B | 方案C | 最优 |
|----------|------|-------|-------|-------|------|
| 常驻内存（菜单栏+Companion） | 25% | 70-140MB | 130-250MB | 90-180MB | **A** |
| 启动/响应延迟 | 20% | 1-3s | 2-4s | 0.5-1s | **C** |
| CPU 空闲开销 | 15% | <1% | 1-3% | <1%-2% | **A** |
| 磁盘占用（安装包） | 15% | ~80-100MB | ~150-200MB | ~0MB | **C** |
| 电池/能耗影响 | 15% | 最低 | 最高（双运行时） | 中等 | **A** |
| 系统启动速度影响 | 10% | 低 | 高 | 极低 | **C** |
| **加权总分** | 100% | **8.5** | **5.5** | **7.0** | **A** |

### 5.2 关键发现

1. **方案A 在性能维度全面最优**：SwiftUI 原生应用的内存和能耗优势巨大（30MB vs 150MB），pkg 打包虽增加体积但换来零依赖部署。唯一短板是启动延迟略高，但可通过预加载和 SEA 优化解决。

2. **方案B 性能代价最高**：Electron "主进程-only" 是一个常见的性能陷阱——即使不创建窗口，Chromium 内核的基线开销不可避免。130-250MB 常驻内存对于"仅显示一个菜单栏图标"的应用完全不合理。双 Node 运行时（Electron 内置 Node + Companion 独立 Node）进一步放大资源浪费。

3. **方案C 性价比最优**：6 人天投入获得接近方案A 的启动速度和系统启动体验，且磁盘占用为零。但双 Node 进程和轮询机制是明显的设计缺陷，若将菜单栏代理合并进 Companion 进程，性能可接近方案A。

4. **隐藏成本：电池续航**：Electron 的 V8 双运行时意味着双倍的定时器、双倍的 GC、双倍的 idle wake，在笔记本上会持续消耗电池。方案A 的 Swift 进程几乎不产生 idle wake，对电池影响最小。

### 5.3 性能 vs 成本权衡

```
性能排序:    A > C > B
成本排序:    C (6人天) > B (8.5人天) > A (13.5人天)
性价比排序:  C > A > B
```

- **若预算充足且追求极致**：选方案A，性能最优，用户体验最佳。
- **若预算有限且接受优化**：选方案C，但务必实施"合并菜单栏代理到 Companion"和"事件驱动替代轮询"两项优化。
- **方案B 不推荐**：除非团队已有 Electron 经验且时间极度紧张，否则其性能代价与收益不成正比。若必须用 Electron，建议直接评估 Tauri 替代。

## 六、跨方案通用优化建议

无论选择哪个方案，以下优化对所有方案均适用：

| 优化项 | 说明 | 预期收益 |
|--------|------|----------|
| **Companion 内存限制** | 启动时添加 `--max-old-space-size=128 --max-semi-space-size=16` | 限制 V8 堆膨胀 |
| **日志自动轮转** | 按天切割，保留 7 天，单文件上限 10MB | 防止磁盘无限增长 |
| **延迟加载非核心模块** | skill 引擎、history store 等按需初始化 | 降低启动内存峰值 |
| **WebSocket 心跳优化** | 空闲时心跳间隔从 5s 延长至 30s | 降低空闲 CPU/网络 |
| **进程优雅关闭** | SIGTERM 时先关闭 WS 连接、保存线程状态、再退出 | 避免状态丢失和端口残留 |
| **开机自启延迟** | 菜单栏/守护进程启动后延迟 5-10 秒再启动 Companion | 减少系统登录阻塞 |
